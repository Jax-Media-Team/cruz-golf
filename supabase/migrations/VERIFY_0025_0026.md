# Migration 0025 + 0026 verification checklist

After pasting `0025_round_pending_finalization.sql` and
`0026_course_library_v2.sql` into the Supabase SQL editor, walk through
this list. Each item is a quick check that the migration did what it
should and nothing it shouldn't.

Both migrations are **non-destructive, idempotent, additive** — re-running
either is safe.

## 0025 — Round lifecycle

### A. Schema sanity

Run in Supabase SQL editor:

```sql
-- Check the new status enum value is allowed.
select conname, pg_get_constraintdef(oid)
  from pg_constraint
 where conrelid = 'public.rounds'::regclass
   and contype = 'c'
   and pg_get_constraintdef(oid) ilike '%status%';
```

Expected output: a check named `rounds_status_check` whose definition
includes `pending_finalization`.

```sql
-- Both new RPCs exist.
select proname from pg_proc
 where proname in ('fn_mark_round_pending', 'fn_resume_round')
   and pronamespace = 'public'::regnamespace;
```

Expected: 2 rows.

### B. Lifecycle transitions (use a real round you own as commissioner)

For a round in your own group:

1. **Live → Pending**
   - On the round page, when all scores are entered, the green banner
     now shows TWO options: "✅ Finalize →" and "Move to awaiting
     finalization".
   - Tapping the second one moves the round to status
     `pending_finalization` without writing settlements.
   - The round drops out of `/dashboard`'s "Live now" bucket and shows
     up under "Awaiting finalization" instead.
   - The amber pill reads "awaiting finalization" everywhere
     (breadcrumb, dashboard, round page header).

2. **Pending → Live**
   - On the round page (status pending), the amber banner offers
     "↩ Resume scoring".
   - Tapping it returns the round to live. Status pill flips back to
     "live" green.

3. **Pending → Finalized**
   - From a pending round, the amber banner's "✅ Finalize now →"
     link works and writes settlements, just like the existing flow.

4. **Score-entry while pending**
   - Open `/rounds/[id]/score-group` on a pending round. Verify you
     can still enter / edit scores. Saves succeed (no RLS denial).

5. **Negative test: from finalized**
   - Try `select fn_mark_round_pending('<finalized-round-id>'::uuid);`
     in SQL editor. Should raise: `Round is already finalized — unfinalize first`.

### C. Records / leaderboards / clubhouse

- Finalized rounds still populate `/records`, `/records/me`,
  `/leaderboards`, `/ledger`. (Their queries already filter on
  `status = 'finalized'`, so they're unaffected.)
- The Clubhouse strip on `/dashboard` shows pending rounds NEITHER as
  live (no green dot) NOR as a streak source. Pending rounds are
  invisible to all signals until they become finalized.
- Existing live rounds continue to appear in the Clubhouse "live now"
  card. Existing finalized rounds continue to feed the streak,
  rivalry, partner, and lifetime signals.

### D. Recoverability

- Archive a pending round (existing UI flow). It should soft-delete
  via `fn_archive_round` and disappear from the dashboard.
- Restore via `/admin` (or `select fn_restore_round('<id>'::uuid);`)
  and verify it returns at the same `pending_finalization` status.

## 0026 — Course library v2

### A. Schema sanity

```sql
-- Check the new columns exist with the right defaults.
select column_name, data_type, column_default
  from information_schema.columns
 where table_schema = 'public' and table_name = 'courses'
   and column_name in ('verification_status', 'submitted_by');
```

Expected: 2 rows. `verification_status` defaults to `'community'`.

```sql
-- The 13 NE FL priority courses are seeded.
select name, city, verification_status
  from public.courses
 where is_template = true
 order by name;
```

Expected: 13+ rows. Each NE FL course has `verification_status =
'placeholder'`. (JGCC may show `verified` if you previously promoted a
template copy; otherwise it'll be `placeholder` until you run
`fn_set_course_verification`.)

```sql
-- The new RPCs exist.
select proname from pg_proc
 where proname in ('fn_set_course_verification', 'fn_set_course_template')
   and pronamespace = 'public'::regnamespace;
```

Expected: 2 rows.

### B. Library rendering

On `/courses` (signed in as a normal user with at least one course):

- The "Course library" section now shows the 12 NE FL placeholder
  courses (filtered to exclude any same-name template that collides
  with your group's own courses).
- Each placeholder card shows:
  - Course name
  - City, state
  - Muted "Awaiting scorecard" pill
  - "📷 Help verify →" button (links to `/courses/import`)
  - **No "Clone into my group" button** — the placeholder shouldn't
    even offer it.

### C. Negative test: clone refusal

Try cloning a placeholder via the SQL editor:

```sql
-- Get a placeholder course id
select id, name from public.courses
 where verification_status = 'placeholder' limit 1;

-- Try to clone it
select fn_clone_course('<placeholder-id>'::uuid);
```

Expected exception: `This course is a placeholder — its scorecard data
hasn't been verified yet. Help us by importing a scorecard photo.`

This protects users from ending up with an empty shell course in
their group.

### D. OCR import flow from a placeholder

1. Navigate to `/courses` and tap a placeholder's "📷 Help verify →".
2. Verify it routes to `/courses/import`.
3. Take a photo OR choose a saved photo from your library
   (PhotoPicker now offers BOTH options on iOS, Android, and desktop).
4. After OCR, the imported course lands in YOUR group as a normal
   course — it does NOT promote the placeholder template. Admin
   moderation is the path for promoting community submissions to
   verified templates.

### E. Admin verification flow

From an admin (platform admin role):

```sql
-- Promote a placeholder to verified once you have its data
select fn_set_course_verification('<course-id>'::uuid, 'verified');

-- Or flag a community submission for review
select fn_set_course_verification('<course-id>'::uuid, 'needs_review');
```

The course's badge on `/courses` updates to "Verified" or
"Needs review" in calm color.

### F. Existing courses still work

- Your group's own courses (non-template) render with the same
  affordances as before. The new column defaults to `'community'`, so
  they all show a calm "Community" badge.
- Existing rounds continue to reference course rows correctly.
  Settlement, leaderboard, and records queries are unchanged.
